map双列集合

总览示意图

map是一个键值对集合，key无序唯一，value不要求有序可重复，主要实现由hashmap、linkedhashmap、treemap、hashtable、concurrenthashmap

  • hashmap：1.8 之前 HashMap 由数组+链表组成的，数组是 HashMap 的主体，链表则是主要为了解决哈希冲突而存在的（“拉链法”解决冲突）；1.8 以后在解决哈希冲突时有了较大的变化，当数组长度大于64并且链表长度大于阈值（默认为 8）时，将链表转化为红黑树，以减少搜索时间（ HashMap 支持 null 键，null的哈希值直接为0）
  • linkedhashmap：LinkedHashMap 继承自 HashMap，底层仍然是基于拉链式散列结构即由数组和链表或红黑树组成；在hashmap的基础上增加了一条双向链表，使得上面的结构可以保持键值对的插入顺序
  • hashtable：线程安全的，所有的方法加上synchronized关键字，其余和hashmap没什么区别（不支持null值null键）
  • treemap：红黑树（自平衡的排序二叉树）
  • concurrenthashmap：数组+链表/红黑树；不支持null值null键；线程安全的（1.8之前使用分段锁，1.8之后使用volatile+CAS或者synchronized）

HashMap

实现原理

JDK1.7之前，HashMap是由数组+链表组成，HashMap通过哈希算法将元素的键映射到桶数组的槽位中。如果多个key映射到同一槽位，就会以链表的形式存储（链表查询为O(n)，冲突严重时查询效率就低了）

JDK1.8做了优化，当桶数组长度大于等于64并且链表的长度大于等于8的时候，就会将链表转换为红黑树（查询为O(logn))，当数量小于等于6的时候就会把红黑树转换回链表

Put操作

调用put的时候：（初始容量是16）

  1. 根据要添加的key来计算在桶数组中对应的位置，计算方式是(数组长度-1) & key的哈希值
  2. 检查对应位置是否为空，如果为空直接把元素插入进去（HashMap的修改次数加1，以便在进行迭代时发现并发修改）
  3. 如果对应位置已经存在元素了，那么检查该位置第一个元素key的哈希值是否和key是否相同，相同直接替换value
  4. 如果不相同，那么遍历链表/红黑树来查找是否有相同的元素，有的话就替换掉
  5. 如果不存在相同的元素，把新建插入链表（jdk1.7 头插法 jdk1.8尾插法）或者红黑树对应的位置
  6. 如果是链表，那么插入之后检查长度是否达到了阈值8，达到并且数组长度>=64则把链表转化为红黑树
  7. 检查负载因子是否达到阈值0.75（负载因子就是键值对的数量/数组长度），如果超过阈值那么进行扩容操作
  8. 扩容操作：
    1. 创建一个新的两倍大小的数组
    2. 把旧数组中数据重新计算哈希值并分配到新数组中
    3. 更新HashMap数组引用和阈值参数

是否线程安全

HashMap不是线程安全的，多线程下存在的问题：

  1. JDK1.7，put的时候采用头插法，那么扩容的时候存在Entry链死循环和put数据丢失的问题

    Entry链死循环示例：

    线程 1 和 线程 2 同时执行 transfer()，步骤拆解：

    线程 1 执行到 Entry<K,V> next = e.next;（保存 e1 的 next 为 e2），然后被 CPU 挂起；

    线程 2 完整执行迁移，将链表变成 e2 → e1 → null，并完成扩容；

    线程 1 恢复执行，此时：

    • 线程 1 中 e = e1，next = e2（旧值）；
    • 执行 e.next = newTable[i] → e1.next = null（新数组该桶此时为空）；
    • 执行 newTable[i] = e1 → 新桶头节点为 e1；
    • 执行 e = next → e = e2；

    线程 1 处理 e2：

    • 执行 next = e2.next → 此时 e2 的 next 已经被线程 2 改成 e1（环形的关键！）；
    • 执行 e2.next = newTable[i] → e2.next = e1；
    • 执行 newTable[i] = e2 → 新桶头节点为 e2；
    • 执行 e = next → e = e1；

    线程 1 进入死循环：

    • 再次处理 e1 时，next = e1.next = null，但 e2.next = e1，形成 e1 ← e2 的环形链表；
    • 后续查询该桶的元素时，会无限遍历 e1 → e2 → e1 → e2...，导致死循环。

  2. JDK1.8，解决了 Entry 链死循环和数据丢失问题，但是存在put数据覆盖的问题

为什么使用红黑树

为什么使用了红黑树？平衡二叉树不可以吗？

  1. 平衡二叉树是追求完全平衡的：任何节点的左右子树高度差不能超过1，这样严格的条件导致每次插入/删除数据几乎都会破坏平衡，就会调整一颗符合条件的平衡树，浪费性能
  2. 红黑树不追求完全平衡：整个树最长路径不会超过最短路径的 2 倍；虽然牺牲了一部分查找的性能效率，但是能够换取一部分维持树平衡状态的成本；红黑树在插入、删除等操作，不会像平衡树那样，频繁着破坏红黑树的规则，所以不需要频繁着调整

扩容机制

为什么每次扩容要是2倍？而且初始的容量必须是2的次方？

  1. 2的次幂，在计算索引的时候使用位运算代替了取模，效率高
  2. 扩容时迁移元素时只需判断 1 个二进制位，降低扩容的计算成本
    • 扩容之后长度-1最左边的二进制位是0还是1
      1. 0的话再哈希位置不变
      2. 1的话原先位置+原先长度

HashMap默认容量是16，扩容是2n

HashTable默认容量是11，扩容是2n+1

怎么计算hash值的

hashmap对key的类型没有做限制，也就是任何类型都可以作为key，那么hash值怎么计算的？

  1. 所有的类都是继承Object，它有一个 hashCode() 方法来计算hash值（int类型）
  2. 对于null的话hash值是写死的，直接为0
  3. 对于基本数据类型，都有自己对应的包装类

通过Object的hashCode()函数计算后，HashMap还会进行二次哈希（扰动），把计算的hashCode的高16位和低16位进行异或

1、为什么？高位扰动的意义？

因为数组长度通常很小（比如初始是 16），直接用 hashCode 取模的话，只有低位参与运算，高位全浪费了。通过 ^ (h >>> 16)，把高位的特征混合到低位，能让哈希值分布更均匀

2、为什么高位扰动使用异或？

或的话存在大量的1，与的话存在大量的0，都会导致哈希值高度集中、分布不均匀

3、为什么(n-1)&hash可以代替取模？

hashmap中桶数组的长度是2的幂次方，比如是n=2^k，n-1可以保证低k位都是1，这时候(n-1)&hash就会把 hash 的高位全部清 0，只保留最后 k 位，结果和取模运算完全相等

ConcurrentHashMap

原理

JDK1.7 分段锁 Segment

JDK1.7底层数据结构采用的是 数组+链表，利用分段锁（可重入）保证线程安全的，ConcurrentHashMap将桶数组分为16段Segment，每个Segment持有独立的锁，每个Segment下面又是一个HashEntry数组，用于存储键值对；当插入/更新/删除等操作，先定位到具体的segment，然后在该segment加锁，这样当一个线程占用锁访问其中一个段数据的时候，其他段的数据也能被其他线程访问

JDK1.8 volatile + CAS 或者 synchronized

JDK1.8底层采用数据+链表/红黑树，利用volatile+CAS/synchronized对头节点进行加锁保证线程安全的

当添加元素的时候：

  1. 首先判断容器是否为空，如果为空使用CAS初始化（volatile修饰）桶数组
  2. 如果容器不为空，那根据存储的元素计算索引判断索引位置是否为空
    1. 如果索引位置为空，那么利用CAS设置该节点
    2. 如果索引位置不为空，那么使用synchronized锁住头节点，然后遍历数据然后在对应位置替换或者新增节点，最后判断是否要转化红黑树
  3. 这样是保证线程安全了的

使用synchronized为什么还要用CAS

两种锁结合保证线程安全是一种权衡的考虑，主要是根据竞争程度来选择具体使用那一种的

put元素的时候如果计算索引没有存放元素，那么使用CAS设置值，因为在设置元素的时候，hash值经过了各种扰动后，发生hash碰撞的概率是较低的，所以可以预测使用较少的自旋来完成具体的设置值

HashTable

原理

底层：数据+链表

线程安全：所有的公共方法都加了synchronized关键字，一个线程访问同步方法，另外一个线程就会阻塞

三个区别

HashMap、ConcurrentHashMap、HashTable

  • HashMap
    1. 线程不完全，效率高点
    2. 可以存储null值null键
    3. 默认容量16，扩容2倍
    4. 给定初始大小时，扩充为离初始大小最近的2的次方
    5. 底层数据+链表/红黑树
  • HashTable
    1. 线程安全，所有的公共方法都加synchronized
    2. 不可以存储null值null键
    3. 默认容量11，扩容2n+1
    4. 给定初始大小时直接使用
    5. 底层数据+链表（基本被淘汰
  • ConcurrentHashMap
    1. 线程安全，分段锁||volatile+CAS/synchronized
