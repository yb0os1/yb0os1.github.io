1、实习过程中有没有高并发的经历

  • 什么是多线程的竞态条件？先查后更新的操作在并发环境下会引发什么问题？
  • 在你的携程规则测试平台项目中，你通过NOT EXISTS原子 SQL 解决了千级用例并发结束时的场景状态一致性问题。如果不使用这条原子 SQL，而是在 Java 代码中先查询数据库是否存在未完成的测试用例，再根据查询结果更新测试场景的状态，会引发哪些问题？你会如何解决这些问题，并设计兜底方案？

2、限流、熔断与重试

  • 请说明什么是限流、什么是重试，二者分别用于解决什么问题？
  • 在你的携程项目中实现了支持多粒度配置的多级动态限流，请说明为什么要设计多级限流？
  • 为什么指数退避重试比固定间隔重试更合理？
  • 当依赖服务响应已经很慢时，系统仍持续进行大规模重试，会引发哪些问题？
  • 什么业务场景适合使用重试机制？什么场景适合使用熔断机制？

3、存储架构设计

  • MySQL 和 ClickHouse 分别适用于哪些业务场景？
  • 你的测试平台为什么要做分层存储设计，将业务状态数据存入 MySQL，执行快照数据存入 ClickHouse？
  • 为什么不把所有数据都存到 ClickHouse 里？
  • 针对实时性要求不高的海量数据，你会通过哪些手段优化提升读取性能？

4、Redis 缓存核心考点

  • Redis 为什么快、执行效率高？
  • 使用 Redis 做缓存，需要考虑哪些核心问题？

5、消息队列

  • 消息队列中自动 ACK 和手动 ACK 的核心区别
  • 在你的项目中，通过解耦消费者线程与业务线程池，将 MQ 异步执行链路的吞吐量提升了 3~5 倍，请你讲一下这套优化方案的核心思路
  • 消息消费的 ACK 提交失败，会引发哪些问题？如果在业务逻辑处理完成之前就提交 ACK，会存在哪些风险？
  • 针对消息队列的重复消费问题，你有哪些解决方案？
  • 什么是事务消息？什么是延时队列？
  • 支付成功的消息丢失了，延时队列到了准备释放库存的阶段，会发生什么问题？
