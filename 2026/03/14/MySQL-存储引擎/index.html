MySQL-存储引擎

执行sql请求过程

MySQL架构可以分为连接层、server层、存储引擎层

执行一条sql语句过程

  1. 连接器，基于TCP协议传输。主要进行建立连接，管理连接、校验用户身份

  2. 查询缓存，查询语句如果命中查询缓存则直接返回，否则继续往下执行。MySQL 8.0 已删除该模块；

  3. 解析器，通过解析器对 SQL 查询语句进行词法分析、语法分析，然后构建语法树，方便后续模块读取表名、字段、语句类型；

    词法分析：根据输入的sql语句识别出关键字和非关键字

    语法分析：根据词法分析的结果，语法解析器会根据语法规则判断这个SQL语句是否满足MySQL的语法

  4. 执行SQL：三个阶段

    1. 预处理器：检查表或字段是否存在、select * 中*代替所有的字段
    2. 优化器：基于查询成本（选择使用哪些索引，以及决定表之间的连接顺序等），选择查询成本最小的执行计划
    3. 执行器：根据执行计划执行SQL查询语句，从存储引擎中读取记录返回给客户端

执行器和存储引擎交互过程比如有主键索引查询、全表扫描、索引下推

主键索引查询：

  1. 执行器第一次查询执行函数，将条件交给存储引擎
  2. 存储引擎通过主键索引B+树判断记录是否存在 存在返回给执行器；不存在上报查询不到的错误；
  3. 执行器从存储引擎读取到记录再次判断是否符合条件 符合交给客户端；不符合跳过记录
  4. 执行器查询过程是while循环，还是需要再次查询，同样执行函数（永远返回-1）执行器退出，结束查询

全表扫描：

  1. 执行器第一次查询，调用函数将条件交给存储引擎

  2. 存储引擎读取表中的第一条记录，符合条件返回给客户端，不符合跳过

    1. Server 层每从存储引擎读到一条记录就会发送给客户端，客户端显示的时候是直接显示所有记录的，是因为客户端是等查询语句查询完成后，才会显示出所有的记录

  3. 执行器查询过程是while循环，还是需要再次查询，执行函数，重复1-2过程，直到存储引擎把表中所有记录读完，然后向执行器返回读取完毕的信息

  4. 执行器收到存储引擎报告的查询完毕的信息，退出循环，停止查询。

索引下推：

为了减少二级索引在查询时的回表操作

age和money建立了联合索引，由于>导致age索引可用，money的索引失效

比如一个查询语句select * from a where age>10 and money=1000

如果不使用索引下推，那么每找到一个age>10的记录都需要进行回表，将完整记录返回给serve层；serve层判断money是否等于1000再决定是否返回给客户端

使用索引下推，找到age>10的记录后，不执行回表操作，而是判断该索引中money列条件是否符合，不符合跳过，符合的话再进行回表操作，将完整记录返回给serve层

MySQL引擎

1、有什么了解？

  • Innodb：默认存储引擎，具有ACID事务支持、行级锁、外键约束；使用高并发的读写操作，支持较好的数据完整性和并发控制
  • MyISAM：具有较低的存储空间和内存消耗，适用于大量读操作的场景。不支持事务、行级锁和外键约束，因此在并发写入和数据完整性方面有一定的限制。
  • Memory：将数据存储在内存中，适用于对性能要求较高的读操作；在服务器重启或崩溃时数据会丢失；不支持事务、行级锁和外键约束

2、为什么默认引擎是Innodb？

InnoDB引擎在事务支持、并发性能、崩溃恢复等方面具有优势

ACID、行级锁、redolog

  • 事务支持：InnoDB引擎提供了对事务的支持，可以进行ACID（原子性、一致性、隔离性、持久性）属性的操作
  • 并发性能：InnoDB引擎采用了行级锁定的机制，可以提供更好的并发性能
  • 崩溃恢复：InnoDB引引擎通过 redolog 日志实现了崩溃恢复，可以在数据库发生异常情况（如断电）时，通过日志文件进行恢复，保证数据的持久性和一致性

3、MyISAM和Innodb对比

  • 事务支持：MyISAM不支持
  • 索引结构：MyISAM是非聚簇索引
  • 锁粒度：MyISAM最小级的锁是表锁
  • count效率：MyISAM保存具体的行数
