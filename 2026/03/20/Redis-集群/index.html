Redis-集群

如何实现服务高可用

实现redis高可用肯定要从redis的多服务节点考虑，比如redis的主从复制、哨兵模式、切片集群

redis集群的优缺点

优点：

  1. 高可用性：最主要的优点，节点之间采用主从复制机制，保证数据的持久性和容错性
  2. 高性能：切片集群把数据分散到多个节点，提高读写能力，增加吞吐量
  3. 扩展性好：可以根据实际动态增加或减少节点

缺点：

  1. 部署和维护复杂
  2. 集群同步问题：当某些节点失败或网络出现故障，集群中数据同步的问题也会体现；数据同步的复杂度和工作量随着节点数量的增加而增加，同步时间也较长
  3. 数据分片限制：将数据分散存储到各个节点，某些操作不能跨节点实现，不同节点之间的一些操作需要额外注意

主从复制

主从复制有完全同步和增量同步两种模式

完全同步

完全同步发生在：

  1. 初次同步：当一个从服务器首次连接主服务器时
  2. 从服务器数据丢失：如果从服务器由于断电等原因导致数据丢失
  3. 主服务器数据发生变化：从服务器长时间没和主服务器同步，导致数据差异太大

完全同步的过程分为：

  1. 从服务器发送sync命令，请求开始同步
  2. 主服务器接收命令后，生成RDB快照并发生给从服务器
  3. 从服务器接收到RDB快照后，情况当前数据集，然后加载RDB快照
  4. RDB快照生成和传输过程中，主服务器把收到的写命令都写入到replication buffer
  5. RDB快照传输完成后，主服务器会把缓冲区的命令发给从服务器，保证数据一致性

增量同步

增量同步发生在：网络连接断开恢复后的同步

增量同步基于psync命令，使用允许id和复制偏移量

具体过程分为：

  1. 从服务器恢复网络后，发送psync命令，此时offset不是-1
  2. 主服务器收到后，使用continue响应告诉从服务器接下来使用增量同步
  3. 主服务器会将断线期间所执行的写命令，也就是增量数据，发送给从服务器

增量数据怎么判断的？

  1. repl_backlog_buffer：环形数组，用于主从服务器断开连接后。主服务器每次执行写命令都会把命令写入到这个缓冲区
  2. offset：标记环形数组同步的进度，主从都有各自的偏移量，主服务器根据自己的偏移量和从服务器传输过来的偏移量判断执行什么同步：
    1. 如果数据不在缓冲区中，那么直接全量同步
    2. 如果数据还在缓冲区中，那么采用增量同步

backlog_buffer默认1M，为了避免网络恢复时，频繁采用全量同步，这个缓冲区的大小应该合适，减少出现从服务器要读取的数据被覆盖的概率

实践优化

全量同步执行较耗时，需要生成RDB文件（磁盘IO），然后发送RDB文件（网络IO）

针对全量同步的优化：

  1. 在主节点启动无磁盘复制，就是不生成RDB文件，而是直接将其全量数据通过网络发送给从节点，避免磁盘IO
  2. 控制单节点上内存占用不要太大，减少全量同步的磁盘和网络IO
  3. 减少全量同步的概率，适当提高repl_backlog缓冲区大小
  4. 主节点同步的压力，如果太多的 slave 节点都去找 master 进行数据同步，那么 master 压力很大；限制一个 master 上的 slave 节点数量，如果实在太多 slave 节点，可以采用主-从-从链式结构，即从节点去找从节点进行数据同步。

哨兵模式

原理

主从架构中是读写分离的，如果主节点挂了那么将没有主节点来处理客户端的写操作了，也没有主节点给从节点进行数据同步了，因此引入了哨兵机制，作用就是实现主从节点故障转移

哨兵节点负责三个事情：监控、选主、通知，监测主节点是否存活，如果主节点挂了就会重新选取一个从节点切换为主节点，并把新主节点信息通知给所有从节点和客户端

选主算法

以下步骤：

  1. 故障节点主观下线：哨兵集群中的每个节点会对redis集群所有节点定时发送心跳包检测节点是否正常，如果在down-after-milliseconds时间内没有回复，认为该节点主观下线
  2. 故障节点客观下线：被标记主观下线不一定是故障，但是如果哨兵集群中超过quorum数量的哨兵节点认为一个redis节点主管下线，那么该redis节点客观下线；如果下线的是从节点操作到此位置，如果是主节点开始故障转移
  3. 哨兵集群选取leader：每个哨兵节点都可以选取为leader，当确认主节点客观下线时，请求其他哨兵节点要求将自己选举为leader
  4. leader决定新主节点：规则
    1. 过滤网络状态不好的节点
    2. 优先选择与主节点断开时间最短的，如果超过指定值那么排除
    3. 判断从节点优先值，越小越优先，0则用不参与选举
    4. 优先值相同，判断offset，越大说明数据越新
    5. 最后判断节点id，越小优先级越高

脑裂现象

脑裂：主节点网络出现问题，与从节点失联但是和客户端网络正常，哨兵发现主节点失联就会从从节点选取一个作为新的主节点，此时集群就有两个主节点

网络好了旧的主节点会被降为从节点，那么这段时间内与客户端的数据就会丢失了

解决方案：

  1. 主节点必须要有至少x个从节点连接，如果小于这个数主节点禁止写数据
  2. 主从数据复制和同步延迟不可以超过x秒，超过那么禁止写数据

切片集群

当缓存数据量大到一台服务器无法缓存时，就需要切片集群方案

把数据分布在不同的服务器上，来降低系统对单主节点的以来，从而提高redis服务的读写能力

切片方案采用哈希槽来处理数据和节点的对应关系，一个切片集群有16384个哈希槽，每个键值对根据key被映射到一个哈希槽：

  1. 根据键值对的key按照CRC16算法得到一个16bit的值
  2. 用16bit的值对16834取模得到对应的哈希槽
