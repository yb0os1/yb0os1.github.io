Redis-场景

缓存

本地缓存

本地缓存指数据存储在本地应用程序或服务器上，通常用于加速数据访问和提高响应速度。通常使用内存为存储介质，利用内存的高速读写

优势：

  1. 访问速度快：本地缓存存储在本地内存，可以，满足频繁访问和及时响应需求
  2. 减轻网络压力：降低对远程服务器的访问次数
  3. 低延迟：本地缓存位于本地设备上，适用对实时性要求高的

不足：

  1. 可扩展性有限：本地缓存可扩展性手硬件资源限制，无法支持大规模数据存储和访问
  2. 服务重启/发布，缓存全部丢失
  3. 集群环境下数据不一致

常见本地缓存框架：

  1. Caffeine
  2. Ehcache
  3. ConcurrentHashMap +自定义淘汰策略

分布式缓存

如redis，将数据存储在多个分布式节点上，通过协同工作提供高性能的数据访问

优势：

  1. 扩展性强：节点可以动态扩展，支持大规模数据存储和访问需求
  2. 数据一致性高：分布式缓存能够保证数据在多个节点之间的一致性
  3. 易于维护：自动化管理方式，能够降低维护成本和管理的复杂性

不足：

  1. 访问速度相对本地缓存慢，数据需要多个节点进行访问和协同
  2. 网络开销大：分布式缓存需要通过网络进行数据传输和协同

缓存穿透

客户端请求的数据在缓存和数据库中都不存在这样缓存永远不会生效，请求一直都会到数据库

解决方案：

  1. 非法请求限制：恶意请求访问不存在的数据时也会缓存击穿，可以在API入口判断请求参数是否合理
  2. 缓存空值或默认值：实现简单维护方便，但是可能会出现短暂数据不一致、额外的内存消耗
  3. 布隆过滤器：写入数据库数据时，使用布隆过滤器做个标记，然后在用户请求到来时，可以通过查询布隆过滤器快速判断数据是否存在；存在误判的可能89

缓存雪崩

大量缓存数据同一时间过期或redis故障宕机时，导致大量请求直接到达数据库，从而导致数据库压力骤增甚至导致数据库宕机

解决方案：

  1. 给不同的key添加随机的ttl （针对key失效）
  2. 利用Redis集群提高服务的可用性（针对redis宕机）
  3. 给缓存业务添加降级限流策略
  4. 给业务添加多级缓存

缓冲击穿

热点数据过期，大量请求访问该热点数据，直接访问数据库，数据库容易被高并发击垮

解决方案：

  1. 互斥锁，保证同一时间只有一个线程访问数据；没有额外的内存消耗、保证一致性、实现简单；但是其他线程等待性能受影响、可能有死锁风险
  2. 逻辑过期，不给热点数据设置过期时间，额外存一个过期时间，但是过期不删除，等到查询的时候如果过期删除重新写入缓存；线程无需等待性能好，但是不保证一致性，额外内存消耗

数据一致性

如何保证MySQL和redis数据一致性？

一般采用旁路缓存策略

  • 读数据，如果缓存不命中，会从数据库加载到缓存中
  • 写数据，先更新数据库，然后再删除缓存

但是旁路缓存策略有个问题，两个操作只要第二个操作出现问题都会出现数据不一致（更新数据库成功，删除缓存失败）

保证最终一致性两种方案：

  1. 引入消息队列，将删除缓存操作的数据加入消息队列，由消费者操作数据
    1. 如果删除缓存失败，那么会重试
    2. 如果成功，就移除
  2. 订阅binlog，更新数据库成功会产生一条日志记录在binlog中，canal模拟成一个MySQL的从节点，向主节点发送dump请求，主节点就会推送binlog给canal，解析后供下游订阅使用

分布式锁

分布式锁用于分布式环境下并发控制的一种机制，用于控制某个资源在同一时刻只能被一个应用所持有

实现原理：

  1. 基于set nx互斥性、ex避免死锁实现不可重入分布式锁，不可重入、无法重试、锁超时失效
  2. 基于hash结构记录线程id和重入次数实现可重入的分布式锁，lua脚本保证原子性
  3. redisson
    1. 利用hash基于线程id和重入次数实现可重入
    2. 基于lua脚本实现原子性
    3. 超时续约机制，当没指定锁过期时间是，利用看门狗机制，每隔一段时间（releaseTime/3）进行锁续约，防止业务未完成锁提前释放
    4. 可重试机制，加锁失败自旋重试；利用信号量和pubsub功能实现等待、唤醒，获取锁失败的重试机制

基于redis的分布式锁在主从集群的问题：

在主节点获取到锁后但是主节点宕机而且还没把数据同步到从节点，那么在新的redis主节点下依旧可以获取到锁，存在多个服务同时获取到锁的可能

解决：

  1. 红锁，让客户端和多个独立的redis节点依次请求申请锁，能够和半数以上节点成功完成加锁操作才认为客户端获取到了锁，否则加锁失败
  2. MultiLock 联锁方案：要求 所有主节点加锁成功，否则立即失败并释放已获取的锁

加红锁的话性能太低了，主从一致性是小概率，如果业务非要保证数据的强一致性采用zookeeper

热点key

什么是热点key？短时间内被高频访问的key是热点key

如何解决热点key问题：

  1. 本地缓存，例如Java的Caffeine地本地缓存，减少Redis的压力，注意设置过期时间
  2. key分片，将热Key拆分成多个子Key，分散到不同的Redis实例上
  3. 读写分离(含主从架构)，读请求分流到多个从节点
  4. 针对热点key做主动数据预热，系统启动时提前加载到缓存

大key

某个key对应的value值占用的内存空间比较大，导致性能下降、内存不足、数据不均衡以及主从同步延迟的问题

大key一般可以认为字符串类型超过1M，集合类型元素数量超过1w 但是不是一成不变的

缺点：

  1. 内存占用高。大key占用过多的内存空间，可能导致可用内存不足
  2. 性能下降。对于大key的读写删等操作，都会消耗更多的CPU时间和内存资源
  3. 阻塞其他操作。比如del删除一个大key可能会导致一段时间内无法响应其他客户端请求，影响系统响应时间和吞吐
  4. 网络阻塞，主从同步延迟：主从同步需要网络传数据
  5. 数据倾斜：

解决：

  1. 对大key进行拆分：将大Key拆分为多个小Key，例如将一个Hash Key拆分成多个Hash Key
  2. 对大key使用异步删除：使用UNLINK（非阻塞）替代DEL，后台异步线程删除，避免影响主线程
  3. 对过期数据进行定期清：堆积大量过期数据会造成大key的产生（hash类型）

布隆过滤器

介绍

布隆过滤器是一种空间高效的概率型数据结构，核心能力是：快速判断一个元素是否可能存在于集合中（存在误判，不存在则 100% 准确）。由位图（bitmap）和k个独立的哈希函数组成

核心操作就是添加和判断

  1. 添加：用 k 个哈希函数，分别计算元素的哈希值，再通过取模运算（模位图大小 m），得到 k 个不同的位图索引（如索引 3、7、12）；将位图中这 k 个索引对应的 bit 位，从 0 设为 1（无论之前是否为 1，只置位不复位）。
  2. 判断：同样用 k 个哈希函数，计算元素的 k 个位图索引；检查这 k 个索引对应的 bit 位：若任意一个 bit 位为 0：元素绝对不存在（直接拦截，无需查底层存储）；若所有 bit 位都为 1：元素可能存在（存在误判，需进一步查底层存储确认）

存在问题：当底层存储的元素过期 / 删除后（如 Redis 中商品 Key 过期、数据库中数据被删除），布隆过滤器中对应的 bit 位依然是 1，会导致：

  1. 过期元素仍被布隆过滤器判定为可能存在，进而触发无效查询（查 Redis/DB 发现不存在）
  2. 长期运行后，位图中 1 的比例越来越高，误判率急剧上升（极端情况所有 bit 位都是 1，布隆过滤器完全失效）

处理无效数据

布隆过滤器过期怎么办？过期之后很多无效的判断怎么处理？

1、定期重建和滚动替换

设定一个重建周期，在后台，根据当前所有有效的、未过期的数据集，构建一个全新的布隆过滤器（B）

一旦 B 构建完成，系统在进行查询时，会先查询 B。如果B 中不存在，则继续查询 A（旧过滤器）。经过一段观察期，确认 B 稳定且准确后，完全废弃 A，让 B 成为主过滤器

优点：简单、可靠，能从根本上消除所有已过期数据造成的误判

缺点：重建过程需要时间和额外的内存空间，且在重建期间（特别是数据量大时），系统需要维护两个过滤器

2、计数型布隆过滤器

标准布隆过滤器的变体，用一个小的计数器数组（例如，每个槽位用 4 个 bit 表示 0-15 的计数）代替了单个位，从而支持删除操作

  1. 插入：对元素对应的所有索引位置，计数器加 1。

  2. 删除/过期：对元素对应的所有索引位置，计数器减 1。

    优势：支持对单个过期元素进行清理，无需整体重建。

    劣势：

    • 内存开销大：计数器替代单个位，通常需要 4 倍以上的内存空间。
    • 假删除风险：如果两个不同的元素共享了某个索引位，其中一个元素的删除会导致该索引位的计数器减小。如果计数器降为 0，就会错误地将另一个元素也删除，导致漏判的风险，这是布隆过滤器设计中要极力避免的
