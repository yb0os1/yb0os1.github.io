MySQL-锁

分类

MySQL锁可分为分为全局锁、表级锁、行级锁

  • 全局锁：通过flush tables with read lock语句将会把整个数据库处理只读状态；主要用于全库逻辑备份，备份期间不会出现数据或表结构的更新，因此备份文件的数据和预期就相同
  • 表级锁：
    • 表锁：通过lock tables对表加锁，表锁会限制别的线程读写，也会限制本线程接下来的读写
    • 元数据锁：对数据库表进行操作时，自动加上MDL，对数据表继续crud操作加MDL读锁，表结构变更加MDL写锁；MDL目的是为了保证当用户对表进行CRUD操作时，防止其他线程对表结构进行变更
    • 意向锁：当插入、删除、更新操作，需要先对表加上意向独占锁，然后对该记录加上独占锁；意向锁目的是为了快速判断表中是否有记录被加锁
    • Auto-inc锁：主键自增锁
  • 行级锁：
    • 记录锁：锁住一条记录；分S锁和X锁，满足读写互斥、写写互斥
    • 间隙锁：只存在可重复读隔离级别，目的是为了解决可重复读下的幻读现象
    • 临建锁：记录锁+间隙锁，锁住的是一个范围和记录本身
    • 插入意向锁：是一个特殊的间隙锁，

表级锁

表锁

表锁会限制别的线程的读写，也会限制本线程接下来的读写

元数据锁

不需要显式使用，对数据库表操作时自动加：

  • CRUD操作，加MDL读锁
  • 表结构变更，加MDL写锁

保证用户执行CRUD操作时，其他线程不能对表结构进行变更

（读写互斥、写写互斥）

什么时候释放？事务提交才释放

申请MDL表会形成一个队列，队列中写锁优先级高于读锁，当MDL写锁等待时，后续所有的CRUD操作都会被阻塞

意向锁

目的是快速判断当前表中是不是有记录被加锁

  • 对记录加共享锁时，需要先对表加上意向共享锁
  • 对记录加独占锁时，需要先对表加上意向独占锁

插入、删除、更新操作先给表加意向独占锁

普通select是无锁 MVCC

但select也是可以加共享锁和独占锁的

意向锁之间不会冲突，但会和表锁发生冲突

AUTO-INC锁

主键自增

该锁不会等待事务提交之后再释放，而是在执行完插入语句之后就立即释放

innodb_autoinc_lock_mode的系统变量用来控制选择用 AUTO-INC 锁，还是轻量级锁

  • 当innodb_autoinc_lock_mode=0，就采用AUTO-INC锁，语句执行结束后才释放锁
  • 当innodb_autoinc lock_mode=2，就采用轻量级锁，申请自增主键后就释放锁，并不需要等语句执行后才释放
  • 当innodb_autoinc_lock_mode =1:
    • 普通insert语句，自增锁在申请之后就马上释放（轻量级
    • 类似insert...select这样的批量插入数据的语句，自增锁还是要等语句结束后才被释放（AUTO-INC

行级锁

S：共享锁，X：独占锁

SS兼容，其余均不兼容

记录锁

锁住的是一条记录，分为S锁和X锁，兼容性如上

  • 一个事务对一条记录加了 S 型记录锁后，其他事务也可以继续对该记录加 S 型记录锁，但是不可以对该记录加 X 型记录锁一个事务
  • 一个事务对一条记录加了 X 型记录锁后，其他事务既不可以对该记录加 S 型记录锁，也不可以对该记录加 X 型记录锁

间隙锁

存在可重复读隔离级别中，为了解决可重复读幻读问题

比如存在id(3,5)的间隙锁，那么其他事务就不可以插入id=4这个记录了

间隙锁之间是兼容的，即两个事务可以同时持有包含共同间隙范围的间隙锁

临键锁

兼容性同样如上

如果一个事务获取了 X 型的 next-key lock，那么另外一个事务在获取相同范围的 X 型的 next-key lock 时，是会被阻塞的。

插入意向锁

特殊的间隙锁，该锁只用于并发插入操作

间隙锁锁住的是一个区间，那么插入意向锁锁住的就是一个点

插入意向锁也属于间隙锁，但两个事务却不能在同一时间内，一个拥有间隙锁，另一个拥有该间隙区间内的插入意向锁（插入意向锁如果不在间隙锁区间内则是可以的）。所以，插入意向锁和间隙锁之间是冲突的。

插入意向锁的生成时机

  • 每插入一条记录，都要看下待插入的位置是否已经被加上间隙锁，如果加上了就会生成一个插入意向锁

update不加索引导致什么？

update的条件没有加索引的话那么就会走全表扫描，这样会对所有的记录加上临键锁，想到与锁住了整个表，变成了全表只读

如何避免？

当 sql_safe_updates 设置为 1 时。

update 语句必须满足如下条件之一才能执行成功：

  • 使用 where，并且 where 条件中必须有索引列；
  • 使用 limit；
  • 同时使用 where 和 limit，此时 where 条件中可以没有索引列；

delete 语句必须满足以下条件能执行成功：

  • 同时使用 where 和 limit，此时 where 条件中可以没有索引列；

MySQL是怎么加行锁的？

加锁的对象是索引，加锁的基本单位是临键锁

加什么锁主要看：走的什么索引、是等值查询还是范围查询、记录存不存在

update x set s=100 where id=2（id是主键）

1、如果id=2存在，那么加上的是记录独占锁

2、如果id=2不存在，那么加上的是间隙锁

可重复读隔离级别下，唯一索引（包含主键索引）和非唯一索引针对当前读的加锁规则：

  • 唯一索引
    • 等值查询
      • 记录存在，在索引树定位到该记录=》临键锁退化成记录锁
      • 记录不存在，在索引树定位到第一个大于和小于该记录=》临键锁退化成间隙锁
    • 范围查询（为什么不一样，临键锁是左闭右开的）
      • 大于等于（存在等值查询）
        • 等值查询的记录在表中=》对该等值的索引退化为记录锁，对大于该等值的索引还是临键锁 ；
        • 等值查询的记录不在表中=》临键锁
      • 小于等于
        • 等值查询的记录在表中，都是临键锁
        • 等值查询的记录不在表中，终止范围前后退化为间隙锁，之前还是临键锁
  • 非唯一索引
    • 等值查询
      • 记录不存在，找到等值的前后两个，临键锁退还为间隙锁
      • 记录存在，因为非唯一索引可能存在值相同的记录，所以是一个扫描的过程，等值前到等值临键锁，等值到等值后一个间隙锁，等值对应的主键记录锁
    • 范围查询
      • 都是临键锁
